## Executive summary

The MVP is **strong on core prep workflow, data model, scoring math, and evaluator-matrix UX** (including nested criteria and live skill focus from the driver). ## technology.md / Convex surface

- **Documented** queries/mutations largely **exist** (`session`, `users`, `evaluations`, `liveEvaluation`).  
- **`listEvaluationsForSession`**: defined and documented; **no app usage found** (grep) — still a **public API surface** if deployed.

- **`getLiveEvalBundle`**: **no server-side auth**; full evaluations load for any client that runs the query with `sessionId`. With `MANAGER_ACCESS_KEY` set, the **page** may set `canManage={false}`, but **data still subscribes** unless gated differently.

## Recommended next priorities (ordered by design debt)

1. **Bias guards (§7)** — lowest hanging fruit in the evaluator matrix: disable manual mark until ≥50% checkpoint **interaction**; optional manager-only discrepancy styling where manager sees peer aggregates.  
2. **Radar or heatmap (pick one thin vertical)** — even a single chart on driver or live-eval would unblock §5.  
3. ~~**Clarify live reveal semantics**~~ — **Done (2026-03-24):** `docs/design-document.md` §3.2 updated (control room vs live-eval, sequential peers, future optional simultaneous mode).  
4. **Convex hardening** (if moving beyond demo): custom queries with auth, or restrict sensitive queries; key driver mutations the same way as live-eval mutations.  
5. **Per-evaluator prep status** — derive from `listEvaluationsForSession`-style data for manager-only UI (without exposing raw rows to evaluators).
