## Executive summary

The MVP is **strong on core prep workflow, data model, scoring math, and evaluator-matrix UX** (including nested criteria and live skill focus from the driver). Large gaps remain versus **design §5 (radar + heatmap)**, **§7 (bias guards)**, and parts of **§3.2 / §6.2 (simultaneous peer reveal, broadcast, per-evaluator prep status)**. **Security is explicitly MVP-trust-model:** several Convex queries/mutations are world-readable or unauthenticated beyond optional page-level `?k=` gates.

## technology.md / Convex surface

- **Documented** queries/mutations largely **exist** (`session`, `users`, `evaluations`, `liveEvaluation`).  
- **`listEvaluationsForSession`**: defined and documented; **no app usage found** (grep) — still a **public API surface** if deployed.  
- **Live-eval mutations** (`setLiveEval*`, `upsertCalibrationMark`): **`assertManagerKey`** when `MANAGER_ACCESS_KEY` is set — good.  
- **`getLiveEvalBundle`**: **no server-side auth**; full evaluations load for any client that runs the query with `sessionId`. With `MANAGER_ACCESS_KEY` set, the **page** may set `canManage={false}`, but **data still subscribes** unless gated differently.
